# RemoteDev v0.16 — Implementation Log

## Overview

This document is the authoritative implementation record for the initial RemoteDev v0.16 codebase. It covers every major component built during the February 2026 implementation sprint: design decisions made, bugs found and fixed, and the rationale behind non-obvious choices. Developers onboarding to the project, or returning after a break, should read this document alongside `docs/PLAN.md` (the original specification) to understand both *what* was built and *why* it was built that way.

RemoteDev is a bridge server that lets you control a Mac development environment from an iPhone via Claude AI. The bridge runs on the Mac, exposes a WebSocket Secure (WSS) endpoint over Tailscale, and proxies between the iPhone (Next.js UI) and a locally-spawned `claude` CLI process. Authentication is HMAC-based; sessions are persistent across transient network drops; all approval decisions are explicit and user-confirmed.

### 1. Project Structure and Package Setup

**File locations:**
- `/home/user/remote-dev/package.json` — root workspace manifest
- `/home/user/remote-dev/bridge-server/package.json` — bridge server workspace
- `/home/user/remote-dev/ui/package.json` — Next.js UI workspace
- `/home/user/remote-dev/ecosystem.config.js` — pm2 process configuration

**What it does:**

The project is organized as an npm monorepo with two workspaces: `bridge-server` (Node.js, no build step, runs directly with `node`) and `ui` (Next.js 14, requires `npm run build` before serving with `next start`). The root `package.json` declares the workspaces and provides top-level convenience scripts.

**Key design decisions:**

- **Monorepo over two separate repos.** Keeping both processes in one repository makes it easy to keep protocol definitions, version bumps, and bin scripts in sync. There is no shared compiled library between them — protocol types are duplicated by design so each workspace has zero coupling at the code level.
- **No TypeScript in bridge-server.** The bridge server is plain ES modules (`.js`) with JSDoc annotations. This eliminates a build step, simplifies debugging, and makes `node --test` the only test runner needed. The UI workspace is TypeScript because Next.js benefits heavily from end-to-end type safety.
- **pm2 for process management.** `ecosystem.config.js` defines both the `bridge` and `ui` processes. A separate `ecosystem.config.local.js` (gitignored, generated by `bin/setup`) overrides secrets and environment variables.
- **Port decision — 7000/7001 instead of 3000/3001.** During review it was noted that ports 3000 and 3001 are the default ports for React dev servers and many other local tooling. Since this project is meant to run on a Mac that is actively being used for development, collisions would be common. Ports 7000 (bridge WSS) and 7001 (Next.js UI) were chosen to avoid these conflicts.

### 2. `auth.js` — Authentication and Brute-Force Protection

**File location:** `/home/user/remote-dev/bridge-server/auth.js`

**What it does:**

Validates HMAC-SHA256 assertion tokens sent by the iPhone client on every WebSocket upgrade request. An assertion is a short-lived, single-use credential that proves possession of the `CLIENT_SECRET` without transmitting it over the wire.

The module exports:
- `validateAssertion(assertion, clientSecret)` — verifies signature, timestamp freshness, and nonce uniqueness
- `checkIPLockout(ip)` / `recordFailure(ip)` / `clearFailure(ip)` — per-IP brute-force tracking

**Key design decisions:**

- **HMAC-SHA256 over JWT.** JWTs require a library and introduce algorithm-confusion attack surface. A simple `HMAC-SHA256(timestamp + "." + nonce, secret)` is auditable in a few lines and has no parser.
- **Nonce replay cache with 30-second TTL.** Each assertion encodes a random nonce. The server keeps a `Map` of seen nonces, expiring entries after 30 seconds (matching the assertion validity window). This prevents replay attacks without requiring a database round-trip on the hot path.
- **`clientSecret` constant-time compare.** Node's `crypto.timingSafeEqual` is used to compare the recomputed HMAC against the provided signature. A naive string comparison would leak secret length via timing side-channel.
- **Per-IP brute-force lockout: 5 failures → 60-second lockout.** After 5 consecutive authentication failures from a single IP, that IP is locked out for 60 seconds. This is implemented with a `Map` keyed by IP, storing `{ failures, lockedUntil }`.
- **Stale-assertion 4003 close is exempt from the lockout counter.** Error code 4003 is sent when the server closes the WebSocket because the assertion timestamp is outside the valid window (e.g., phone clock skew, or a legitimate reconnect with a cached assertion after a token rotation). This is not a brute-force attempt — it is a benign desync. Incrementing the lockout counter for 4003 would cause the phone to lock itself out during a normal `bin/reload` cycle. The 4003 close also signals the UI to discard its cached token and re-fetch from `/api/token`, which self-heals without user intervention.

**Bugs found and fixed:**

None during implementation. The nonce TTL cleanup runs on insertion (lazy expiry) to avoid the overhead of a `setInterval`.

### 3. `session-manager.js` — Global Session Registry

**File location:** `/home/user/remote-dev/bridge-server/session-manager.js`

**What it does:**

A global singleton (module-level state, imported once and shared across all request handlers) that owns:
- The `Map<sessionId, Session>` of all live sessions
- The current `controllerEpoch` — an integer incremented each time a new controller takes ownership
- `mintEpoch()` / `validateEpoch(epoch)` — epoch issuance and validation
- `setController(ws, sessionId)` / `releaseController()` — controller lifecycle
- `activePids()` — returns an array of PIDs of all currently running `claude` processes (used by orphan-scan)

**Key design decisions:**

- **`controllerEpoch` is stored on the session manager, NOT per-session.** This was an explicit decision documented in the spec. There is only ever one controller at a time in v1; the epoch is a global sequence number. Storing it per-session would be wrong because a reconnecting client to a *different* session would not get a fresh epoch check. The global epoch ensures that any stale WebSocket from a previous connection immediately fails its first mutating message and is closed with 4403.
- **Singleton pattern via module-level state.** Node's module cache guarantees that `require()`/`import` of the same file returns the same object. No dependency injection framework is needed — the module itself is the singleton.

**Bugs found and fixed:**

**`activePids()` was calling `.filter()` and `.map()` on a `MapIterator`.**

`Map.prototype.values()` returns a `MapIterator`, not an array. `MapIterator` does not have `.filter()` or `.map()` methods — those belong to `Array.prototype`. The original code:

```js
// WRONG — MapIterator does not have .filter()/.map()
return this.sessions.values()
  .filter(s => s.childPid != null)
  .map(s => s.childPid);
```

Fixed by spreading the iterator into an array first:

```js
// CORRECT — spread to array before calling array methods
return [...this.sessions.values()]
  .filter(s => s.childPid != null)
  .map(s => s.childPid);
```

This bug would have caused a `TypeError` at runtime the first time `orphan-scan.js` called `activePids()`.

### 4. `db.js` — SQLite Persistence Layer

**File location:** `/home/user/remote-dev/bridge-server/db.js`

**What it does:**

Wraps `better-sqlite3` and provides all database operations for session metadata, transcript storage, audit logging, and client error telemetry. All writes are synchronous (better-sqlite3's model), but batched to avoid per-event write amplification.

**Schema:**

| Table | Purpose |
|---|---|
| `sessions` | Session metadata: id, state, created_at, ended_at, transcript_bytes |
| `transcript` | Ordered output events: session_id, seq, role, content, ts |
| `audit_log` | Security events: ip, event_type, detail, ts |
| `client_errors` | UI-reported errors: session_id, message, stack, ts |

**Key design decisions:**

- **WAL mode enabled at open.** `PRAGMA journal_mode=WAL` is set immediately after the database connection is opened. WAL mode allows concurrent readers while a write is in progress, which matters because gap recovery reads transcript rows while new rows are being written.
- **30-day retention pruning + VACUUM after pruning.** A scheduled prune job (runs at startup and every 24 hours) deletes rows older than 30 days from all tables. After deletion, `PRAGMA wal_checkpoint(PASSIVE)` is run followed by `VACUUM` to reclaim disk space. The VACUUM is run *after* pruning, not continuously, to avoid blocking writes.
- **Batched writes: 100ms window, up to 50 events per batch.** Transcript events arrive rapidly during active `claude` output. Writing each event synchronously would serialize the event loop. Instead, events are accumulated in a buffer and flushed either when 100ms has elapsed or when 50 events have accumulated, whichever comes first. This reduces write syscalls by ~50x during heavy output.
- **Per-session transcript byte budget tracked in `sessions.transcript_bytes`.** Each session accumulates a running byte count of transcript content. This is checked against the 50MB offline budget (see `session.js`). The count is updated in the same batch write transaction as the transcript rows, so it is always consistent.

**Bugs found and fixed:**

None during implementation.

### 5. `ndjson-framer.js` — Fault-Tolerant Line Framer

**File location:** `/home/user/remote-dev/bridge-server/ndjson-framer.js`

**What it does:**

Accepts arbitrary `Buffer` or `string` chunks from the `claude` process stdout stream and emits complete newline-delimited JSON lines. Because Node.js streams deliver data in arbitrary chunk sizes (a single line may arrive in multiple chunks, or multiple lines may arrive in a single chunk), a stateful framer is required.

**Key design decisions:**

- **1MB maximum line size.** If a partial line accumulates more than 1,048,576 bytes without a newline, the framer enters truncation recovery mode: it discards the oversized buffer and waits for the next newline boundary before resuming normal operation. This prevents unbounded memory growth if `claude` produces malformed output.
- **Truncation recovery — not a hard crash.** When a line is truncated, the framer emits a special `{ type: "framer_truncation" }` synthetic event so that `session.js` can record the anomaly and potentially trigger degraded mode, rather than silently dropping output or crashing the bridge.
- **Handles chunks split across `push()` calls.** The internal buffer is a string accumulator. On each `push()`, the new chunk is appended and the buffer is scanned for `\n` characters. Complete lines are emitted; the remainder stays in the buffer for the next push.

**Bugs found and fixed:**

None during implementation.

### 6. `path-validator.js` — Safe Path Validation with ReDoS Protection

**File location:** `/home/user/remote-dev/bridge-server/path-validator.js`

**What it does:**

Validates that a file path requested by `claude` (e.g., for a read or write tool call) is within the configured `ALLOWED_ROOT` directory. This is a security-critical module — a bypass here would allow `claude` to read or modify arbitrary files on the Mac.

**Key design decisions:**

- **Ancestor `realpathSync` + `path.relative()` tail computation.** The naive approach — `realpathSync(requestedPath)` and then checking if it starts with `realpathSync(ALLOWED_ROOT)` — fails when the requested path contains symlinks that resolve to a path *longer* than the original. For example, if `ALLOWED_ROOT` is `/projects/myapp` and a symlink inside it resolves to `/var/folders/...`, the resolved path does not start with the allowed root, even though the symlink is within it.

  The correct approach (specified in the plan) is:
  1. Walk the path components one by one, resolving only the *existing ancestor* portion with `realpathSync`.
  2. Append the remaining (not-yet-existing) tail components with `path.join`.
  3. Use `path.relative(allowedRoot, resolvedPath)` and check that the result does not start with `..`.

  This correctly handles symlinks that resolve to longer paths because the anchor point (the deepest existing ancestor) is what gets resolved, not the full requested path.

- **`re2` with 50ms timeout fallback for ReDoS protection.** Path validation uses a regular expression to reject obviously malicious path components (e.g., null bytes, path traversal sequences). Regular expressions on untrusted input can be vulnerable to ReDoS (Regular Expression Denial of Service), where a crafted input causes catastrophic backtracking. The module attempts to import `re2` (a Node.js binding for Google's RE2 library, which has linear-time guarantees). If `re2` is not installed (it is a native addon and may not build on all platforms), the module falls back to Node's native `RegExp` but wraps the match in a `Promise.race` with a 50ms timeout. If the regex does not complete within 50ms, the path is rejected as invalid. This is the approach specified in the plan and provides defense-in-depth without hard-depending on a native addon.

**Bugs found and fixed:**

None during implementation. The `re2` fallback path is exercised by the `test/redos.test.js` timing test.

### 7. `session.js` — Full Session Lifecycle

**File location:** `/home/user/remote-dev/bridge-server/session.js`

**What it does:**

The most complex module in the bridge. A `Session` object owns the entire lifecycle of one user session: spawning `claude`, routing messages, handling disconnects and reconnects, enforcing approval flows, managing backpressure, and tearing everything down cleanly on `endSession()`.

**Session states:**

```
IDLE → RUNNING → AWAITING_APPROVAL → RUNNING (on approval)
                                    → DISCONNECTED (on ws drop during approval)
RUNNING → DISCONNECTED → RUNNING (on reconnect within TTL)
DISCONNECTED → (TTL expires) → session ended, DENY sent if was AWAITING_APPROVAL
```

**Key design decisions:**

- **`preDisconnectState` preserved on disconnect — no auto-deny.** When the WebSocket drops while a session is `AWAITING_APPROVAL`, the session does *not* automatically deny the pending tool call. Instead, `preDisconnectState` stores `"AWAITING_APPROVAL"` and the pending request. If the client reconnects within the TTL, `onReconnect()` restores the `AWAITING_APPROVAL` state and the approval modal is re-presented. The TTL-expiry timer fires a DENY only if the session remains `DISCONNECTED` for the full TTL duration. This prevents spurious denials on brief network drops.

- **TTL-expiry timer fires DENY only while DISCONNECTED.** The timer callback checks `this.state === "DISCONNECTED"` before sending the deny. If the client has already reconnected (state is back to `AWAITING_APPROVAL` or `RUNNING`), the timer is a no-op.

- **`onReconnect()` restores `AWAITING_APPROVAL` within TTL.** When a client reconnects and provides a valid session ID:
  1. The new WebSocket is attached.
  2. If `preDisconnectState === "AWAITING_APPROVAL"`, state is restored to `AWAITING_APPROVAL`.
  3. The approval modal data is resent to the client so it can re-present the prompt.
  4. The TTL timer is cancelled.

- **Exactly-once control response via `lastRespondedRequestId`.** The bridge must send exactly one `control_response` (approve or deny) per tool-call request, even if the client sends duplicate approval messages (e.g., due to reconnect + resend). `lastRespondedRequestId` stores the ID of the most recently responded request. If an incoming approval message has the same ID, it is silently dropped.

- **Backpressure: 30s warn / 60s `ws.close(1011)` — session stays `RUNNING`.** If the client's WebSocket send buffer grows beyond the backpressure threshold (indicating the client is not reading fast enough), a `bridge_warning` of subtype `backpressure` is sent after 30 seconds. If the condition persists for 60 seconds, the WebSocket is closed with code 1011 (server error). Critically, *the session itself stays in `RUNNING` state* — `claude` continues executing. The client is expected to reconnect and receive the buffered output via gap recovery.

- **Parse degraded mode: 5 errors → raw stream; 20 raw lines → `endSession()`.** If the NDJSON framer emits 5 consecutive parse errors (malformed JSON from `claude`), the session enters "degraded mode": subsequent output is forwarded to the client as raw strings rather than parsed objects. If 20 consecutive raw lines arrive in degraded mode without a valid JSON line resetting the counter, `endSession()` is called. This prevents the bridge from hanging indefinitely on a broken `claude` process.

- **50MB offline transcript budget per-session-total, reset only in `endSession()`.** While the client is disconnected, transcript events are buffered in memory. The buffer has a hard cap of 50MB (checked against `db.js`'s per-session byte count). Once the cap is reached, new transcript events are dropped (with a warning logged). The budget counter is reset only when `endSession()` is called — not on reconnect — so a client cannot circumvent the cap by repeatedly disconnecting and reconnecting.

- **Ring buffer (200 entries) + SQLite gap recovery.** The session maintains an in-memory ring buffer of the last 200 transcript entries. On reconnect, the client sends its `lastSeq`; if the gap fits within the ring buffer, events are replayed from memory. If the gap is larger than 200 entries, gap recovery falls back to a SQLite query for the missing range.

**Bugs found and fixed:**

None specific to `session.js` beyond the `activePids()` bug in `session-manager.js` (documented above).

### 8. `wal.js` — WAL Checkpoint Scheduling

**File location:** `/home/user/remote-dev/bridge-server/wal.js`

**What it does:**

Schedules SQLite WAL (Write-Ahead Log) checkpoints after sessions end, keeping the WAL file from growing unboundedly.

**Key design decisions:**

- **PASSIVE checkpoint, not TRUNCATE.** SQLite supports several checkpoint modes. `TRUNCATE` resets the WAL file to zero bytes but requires an exclusive lock, which would block all readers and writers during the checkpoint. `PASSIVE` checkpoints as many frames as possible without waiting for readers to finish — it is non-blocking and safe to run at any time. WAL files may not shrink to zero in passive mode, but they stay bounded in practice.

- **Non-blocking via `setImmediate` after `endSession`.** The checkpoint is not run synchronously inside `endSession()`. Instead, `setImmediate(() => maybeCheckpoint())` defers it to the next iteration of the event loop, after all synchronous cleanup has completed. This ensures that the session teardown path (freeing the WebSocket, clearing timers, etc.) is not delayed by the checkpoint I/O.

- **Rate-limited to once per 5 minutes.** Checkpoints are not free. Running one after every single session would be wasteful if sessions are short and frequent. A `lastCheckpointAt` timestamp is maintained; `maybeCheckpoint()` returns early if fewer than 5 minutes have elapsed since the last checkpoint.

- **Defers if active sessions remain.** If `sessionManager.activeSessions().length > 0` when `maybeCheckpoint()` runs, the checkpoint is skipped. Active sessions generate WAL frames continuously; checkpointing mid-session can cause I/O spikes. The checkpoint will run after the next session ends.

**Bugs found and fixed:**

None during implementation.

### 9. `orphan-scan.js` — Orphaned Claude Process Detection

**File location:** `/home/user/remote-dev/bridge-server/orphan-scan.js`

**What it does:**

Periodically scans for `claude` processes that are still running but whose parent session has ended (or whose bridge crashed). Orphaned processes waste CPU and may accumulate costs. The scanner kills orphans with `SIGTERM` (followed by `SIGKILL` after 5 seconds if still alive).

**Key design decisions:**

- **Uses `REMOTEDEV_SESSION_ID` environment variable to identify owned processes.** When the bridge spawns a `claude` process, it injects `REMOTEDEV_SESSION_ID=<sessionId>` into the child's environment. The orphan scanner identifies processes by checking for this variable — not just by process name. This avoids false positives: if the user is running `claude` themselves in a terminal, that process will not have `REMOTEDEV_SESSION_ID` set and will not be killed.

- **`/proc/environ` on Linux, `ps -E` on macOS.** Reading environment variables of another process requires different mechanisms on different OSes. On Linux, `/proc/<pid>/environ` is a null-byte-delimited list of `KEY=VALUE` pairs readable if the process is owned by the same user. On macOS, `ps -E -o pid,command -p <pid>` includes environment variables in the output. The scanner detects the platform and uses the appropriate method.

- **Skips processes less than 10 seconds old.** There is a race window between when the bridge spawns `claude` and when the session manager records the new PID. A newly spawned process that is not yet registered would appear to be an orphan. The 10-second age filter eliminates this false-positive window.

- **Runs at startup + every 5 minutes.** The startup run catches any orphans from a previous bridge crash. The 5-minute interval bounds the worst-case orphan duration.

**Bugs found and fixed:**

None during implementation.

### 10. `startup-checks.js` — Pre-Start Validation

**File location:** `/home/user/remote-dev/bridge-server/startup-checks.js`

**What it does:**

Runs a set of validation checks when the bridge starts and logs warnings or refuses to start if critical conditions are not met.

**Checks performed:**

| Check | Warn threshold | Refuse threshold |
|---|---|---|
| TLS cert expiry | 14 days, 7 days, 3 days | < 24 hours |
| `claude` CLI version | — (logs version) | Not found in PATH |
| Disk space | < 500MB free | — |

**Key design decisions:**

- **Three-tier cert expiry warning.** Certificates expire silently if not monitored. The bridge warns at 14 days ("renew soon"), 7 days ("renew now"), and 3 days ("URGENT"). At less than 24 hours remaining, the bridge refuses to start — a TLS cert that expires mid-session would drop all connections without warning.

- **`claude` version check is informational, not blocking.** The bridge logs the installed `claude` version on startup. If `claude` is not found in `PATH`, the bridge refuses to start (it is a hard dependency). But an unexpected version produces only a warning — the bridge does not enforce a minimum version at startup because the version string format may change.

- **Disk space check is a warning, not a block.** If disk space drops below 500MB, the bridge logs a warning. It does not refuse to start, because the disk might be nearly full for unrelated reasons and refusing to start would prevent the user from even connecting to fix it. SQLite writes will fail naturally if the disk fills completely.

**Bugs found and fixed:**

None during implementation.

### 11. `index.js` — Main Bridge Server

**File location:** `/home/user/remote-dev/bridge-server/index.js`

**What it does:**

The entry point. Sets up the WSS (WebSocket Secure) server, handles the HTTP upgrade, orchestrates authentication, routes all Client→Server messages to the appropriate session handlers, and manages the per-IP unauthenticated connection limit.

**Key design decisions:**

- **WSS server setup.** The server uses Node's `https` module with the TLS certificate and key paths from environment variables. The `ws` library's `WebSocketServer` is attached to the HTTPS server, inheriting TLS.

- **Per-IP unauthenticated connection limit: 10.** Before authentication completes, an IP can hold at most 10 simultaneous WebSocket connections. Connections beyond 10 are rejected immediately. This prevents a trivial resource exhaustion attack (opening thousands of WebSocket connections without authenticating).

- **5-second authentication timeout.** After the WebSocket upgrade, the client has 5 seconds to send its authentication assertion. If no assertion arrives within 5 seconds, the connection is closed with code 4001 (auth timeout). This prevents connections from idling in the pre-auth state indefinitely.

- **Protocol v1 envelope routing.** All messages conform to the envelope schema `{ v: 1, type: string, payload: object }`. Unknown `type` values are logged and ignored (forward-compatible). Messages with `v !== 1` are rejected with an error response.

- **Controller epoch validation on mutating messages.** Before processing any message that mutates session state (send instruction, approve/deny, release controller), `index.js` checks that the message's `controllerEpoch` matches the current epoch from `session-manager.js`. A mismatch indicates a stale connection (e.g., a tab that was open before a `bin/reload`). The connection is closed with code 4403 (epoch mismatch). This is a hard close — not a warning — because a stale controller issuing approvals would be a security issue.

- **All C→S message handlers.** The full set of client-to-server messages handled:
  - `auth` — assertion validation, epoch minting, session join or create
  - `send_instruction` — forward user text to claude
  - `control_response` — approve or deny a pending tool call
  - `release_controller` — voluntarily release controller role
  - `ping` — keepalive, responds with `pong` + server timestamp
  - `client_error` — UI-reported error, written to `client_errors` table

**Bugs found and fixed:**

None during implementation.

### 13. Next.js UI

**File locations:**
- `/home/user/remote-dev/ui/app/api/token/route.ts` — HMAC assertion endpoint
- `/home/user/remote-dev/ui/hooks/useWebSocket.ts` — WebSocket lifecycle hook
- `/home/user/remote-dev/ui/app/page.tsx` — Main chat page
- `/home/user/remote-dev/ui/components/ApprovalModal.tsx` — Tool approval modal
- `/home/user/remote-dev/ui/components/ChatMessage.tsx` — Message renderer
- `/home/user/remote-dev/ui/components/WarningBanner.tsx` — Warning display
- `/home/user/remote-dev/ui/lib/protocol.ts` — Zod schemas

**What it does:**

The iPhone-facing Next.js application. Renders the chat interface, manages the WebSocket connection lifecycle, and presents approval modals for tool calls.

**`/api/token` route:**

- Declared `export const dynamic = "force-dynamic"` — disables Next.js static caching so every request generates a fresh assertion.
- Computes `HMAC-SHA256(timestamp + "." + nonce, CLIENT_SECRET)` server-side and returns the signed assertion.
- **Never returns the raw `CLIENT_SECRET`** — the secret stays on the server. The iPhone receives only a time-limited, single-use token.

**`useWebSocket.ts` hook:**

- **4003 close handler: clears cached token + re-fetches without lockout increment.** When the bridge closes with code 4003 (stale assertion), the hook discards the cached token, fetches a fresh one from `/api/token`, and reconnects. Because the bridge exempts 4003 from the lockout counter, this self-healing loop cannot trigger a lockout.
- **1011 close handler: jittered reconnect 200–1000ms.** When the bridge closes with code 1011 (backpressure eviction), the hook reconnects after a random delay between 200ms and 1000ms. The jitter prevents thundering-herd if multiple tabs reconnect simultaneously.
- **`controllerEpoch` stored in hook state, included in all mutating messages.** The epoch received in the `auth_ok` message is stored in a React ref and appended to every outgoing mutating message. Stale-epoch detection on the bridge side closes the connection with 4403, which is treated as a terminal error (no automatic reconnect — the user must refresh).
- **`visibilitychange` reconnect.** When the iPhone wakes from sleep or the Safari tab becomes visible again, the hook checks if the WebSocket is still open and reconnects if not. This handles the common scenario where iOS suspends the tab during a long session.
- **Periodic pings with `lastAck` tracking.** The hook sends a `ping` message every 30 seconds and records the server's `pong` response time. If no `pong` arrives within 10 seconds of a `ping`, the connection is considered dead and the hook reconnects. `lastAck` is included in reconnect context so the bridge can compute the gap.

**`page.tsx`:**

- **`state_sync` overwrites local state.** The bridge sends a full `state_sync` message on auth (and on `resync_required`). The UI overwrites its local state with the server's authoritative version rather than attempting to merge. This keeps the UI trivially consistent.
- **`resync_required` handling.** If the bridge determines that the client's `lastSeq` is too far behind for ring-buffer gap recovery, it sends `resync_required`. The UI clears its local transcript and requests a full sync.
- **`session_busy` banner.** If the client connects but another controller is already active, a banner is shown ("Session is controlled by another device").
- **`bridge_warning` banners.** Subtypes `backpressure`, `cert_expiry`, `disk_space`, and `parse_degraded` each render a distinct warning banner with appropriate severity and copy.
- **React error → `client_error` WS message.** A top-level React error boundary wraps the UI. If an unhandled React error occurs, the error boundary catches it, formats the message and stack, and sends a `client_error` message over the WebSocket. This telemetry is written to the `client_errors` SQLite table on the bridge, making UI bugs visible in server logs.

**`ApprovalModal`:**

- **Bash tool requires typing "APPROVE" exactly.** For tool calls that involve the Bash tool (arbitrary shell execution), the approval modal requires the user to type the string `APPROVE` (all caps) into a text field before the approve button becomes enabled. **File locations:**

- `/home/user/remote-dev/bin/reload` — pm2 restart with readiness poll
- `/home/user/remote-dev/bin/test-tailnet` — tailnet test suite / phase gate

**`bin/setup`:**

An interactive shell script that walks the operator through Phase 0:
1. Checks for Tailscale and prompts to install if missing.
2. Runs `tailscale cert <hostname>` to obtain TLS certificate.
3. Generates a cryptographically random `CLIENT_SECRET`.
4. Stores the secret in macOS Keychain via `security add-generic-password`.
5. Generates `ecosystem.config.local.js` with the resolved paths and secret reference.
6. Prompts the operator to store the secret in iPhone Safari's password manager.

**`bin/reload`:**

```bash
pm2 restart bridge ui --update-env
```

Critically: uses `pm2 restart`, **not** `pm2 reload`. `pm2 reload` performs a rolling zero-downtime reload, which would leave two generations of the bridge running simultaneously for a brief window. During that window, a client could authenticate against the old process and then have its WebSocket handed to the new process, which would have a different `controllerEpoch`. `pm2 restart` does a clean stop-and-start, ensuring a single-epoch transition. After restart, the script polls the bridge's health endpoint until it responds (up to 30 seconds) before exiting, so callers know when the bridge is ready.

**`bin/test-tailnet`:**

A combined automated + manual test suite that serves as the Phase 1b phase gate. Automated tests include: TLS handshake verification, assertion auth round-trip, ping/pong latency measurement, 1011 backpressure eviction and reconnect. Manual prompts guide the operator through: sending an instruction from the iPhone, approving a tool call, and verifying gap recovery after a forced disconnect. The script exits non-zero if any automated check fails.

### Fix: NdjsonFramer `line_too_long` not firing on single-push oversized lines

**Commit:** `af19926`

**Bug:** `ndjson-framer.js` had a flaw in the `line_too_long` detection path. When an oversized line (>1MB) arrived as a single `push()` call (i.e., the entire oversized line in one chunk, not accumulated across multiple calls), the truncation logic did not fire. Only the accumulation path was guarded.

**Fix:** Added the size check at the top of `push()` before the append-and-scan loop, so an incoming chunk that already exceeds the limit triggers truncation recovery immediately, regardless of whether the buffer was previously empty.

**Why this matters:** The 1MB cap is a memory-safety guard. Without this fix, a single malformed `claude` output line could exhaust heap.

### Fix: Duplicate React instance in production build

**Commit:** `1b86fa4`

Running `npm run build` in the `ui/` workspace failed with a "duplicate React" error. The root cause: the monorepo root `package.json` declared `react` and `react-dom` as dependencies (to satisfy pm2's workspace resolution), and the `ui/package.json` also declared them. npm hoisted both, but the versions resolved differently, producing two React instances on the module graph.

**Fix:** Removed `react` and `react-dom` from the root `package.json` (they belong only in `ui/`). Added `"dedupe": true` logic via `npm dedupe` in the lock file. Also added `alias` entries in `ui/next.config.js` to pin React resolution to the `ui/node_modules/react` copies:

### Fix: Use `tailscale serve` for UI HTTPS instead of a custom Node server

**Commit:** `7d2907f`

**Problem:** Next.js's `next start` does not support TLS natively. The initial plan assumed a custom Node.js HTTPS wrapper around `next start`. During implementation, it became clear this approach had two issues:
1. The custom wrapper would need to duplicate TLS cert loading logic already present in the bridge.
2. pm2's process management would need to be aware of the wrapper script, not `next start`, complicating restarts.

**Solution:** Use `tailscale serve` as a zero-code HTTPS reverse proxy:

```
iPhone Safari  →  https://<host>:7000  →  tailscale serve  →  http://localhost:17000 (Next.js)
```

`tailscale serve` handles TLS termination using the same Tailscale-issued cert that the bridge uses for WSS. This means:
- No custom TLS code in the UI process.
- Next.js binds only to `127.0.0.1:17000` (not reachable from the network directly).
- The Tailscale ACL grants the iPhone access to port 7000 (HTTPS via tailscale serve); direct access to 17000 is not needed.

**Changes:**
- `ecosystem.config.js`: Added `remotedev-tailscale-serve` pm2 app (`tailscale serve --https=7000 http://localhost:17000`). Changed UI `HOST` from `0.0.0.0` to `127.0.0.1` and port from `7000` to `17000`.
- `bin/setup`: Added Step 3 to configure `tailscale serve` idempotently (checks if already configured before running). Renumbered subsequent steps. Added the iPhone Safari URL to the setup summary output.
- `bin/tailscale-serve`: New bin script to manage the `tailscale serve` configuration independently (start, stop, status subcommands).

**Architectural decision — port numbering:**
- `7000`: Public HTTPS port (via Tailscale — what the iPhone sees)
- `7001`: Bridge WSS (direct TLS, reachable on Tailnet)
- `17000`: Internal Next.js HTTP (localhost only, not exposed)

### Spawn Claude under PTY (node-pty) so stream-json flushes

**Date:** February 2026

The docs state that `--output-format stream-json` is intended for pipe and programmatic use (e.g. "works seamlessly with Unix pipes"). In practice, when the bridge spawned `claude` with `stdio: ['pipe', 'pipe', 'pipe']`, the CLI did not flush stdout — the bridge received no data until the process exited, so the UI showed "Running" but no output. This matches reported behavior in anthropics/claude-code#12965 and related issues: output is buffered when stdout is not a TTY. The suggested fix there was to make the CLI stream when stdout is a pipe; until that is fixed upstream, we spawn `claude` under a pseudo-TTY (via `node-pty`) so the process sees a TTY and uses line-buffered stdout. No CLI flag or documented env var was found to force unbuffered output. The bridge now uses `pty.spawn()` for the claude process only; caffeinate and all other logic are unchanged. Kill path and `killAllSessions` support both PTY (`.kill()`) and legacy ChildProcess (`process.kill(-pid)`).

### PTY spawn failure: spawn-helper not executable + diagnostics

**Date:** February 2026

On macOS, node-pty does not exec the target directly; it spawns a **spawn-helper** binary (under `prebuilds/darwin-*/spawn-helper`) which then execs the real process. Diagnostics showed `ptyHelperMode: "100644"` and `ptyHelperExecutable: false` — the helper had no execute bit, so `posix_spawn` failed with "posix_spawnp failed." Some npm installs or prebuild packs ship the helper with mode 0644. **Fix:** Before the first PTY spawn attempt, the bridge calls `ensurePtyHelperExecutable()`: on darwin it stat's the helper and, if it exists but is not executable, does `chmod(helperPath, 0o755)` and logs one line. No restart required for that fix to take effect for the next spawn; after any other bridge code or config change, run **`npm run reload`** (or `./bin/reload`) so the bridge and UI restart with the new code.

**Observability:** When PTY spawn fails, the bridge logs a `ptySpawnDiagnostics` object (claude path/stat/shebang, `claudeSpawnSyncOk`, spawn-helper path and executable bit, process uid/gid, cwd). That made the 0644 helper the obvious cause.
