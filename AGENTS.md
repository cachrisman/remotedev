# AGENTS.md — RemoteDev

Version: 1

Guidelines for Claude Code and other AI agents working in this repository.

## Default workflow: investigate → plan → confirm → implement

For any non-trivial task (bug fix, feature, refactor), follow this sequence:

1. **Investigate** — read the relevant code, understand the root cause or scope.
2. **Plan** — determine the best approach.
3. **Present findings concisely** — share what you found and what you propose to do, as briefly as possible. No padding, no re-stating what the user already said, no lengthy preamble. Just: root cause (if a bug), proposed fix or approach, and any tradeoffs worth flagging.
4. **Wait for approval** — do not implement until the user confirms, requests changes, or explicitly says to proceed.

The user may override this by saying "just do it" or similar — in that case, proceed directly to implementation.

**When reporting findings, be terse.** One short paragraph or a small bullet list is almost always enough. Err toward saying less, not more.

## What this project is

RemoteDev is a Mac-hosted bridge that lets you control a local development environment from an iPhone via Claude AI. It has two processes:

- **`bridge-server/`** — Node.js WebSocket Secure (WSS) server on port 7001. Spawns `claude -p --output-format stream-json`, routes tool approvals, and persists transcripts to SQLite.
- **`ui/`** — Next.js 16 / React 19 chat UI served via Tailscale at `https://<host>:7000`. The iPhone talks to this.

Both are orchestrated by pm2 using `ecosystem.config.js` + `ecosystem.config.local.js` (gitignored, generated by `bin/setup`).

## Non-negotiable rules

### Security — do not violate these

1. **Never expose `BRIDGE_AUTH_TOKEN` or `REMOTEDEV_CLIENT_SECRET` to the client.** The `/api/token` route returns only a signed HMAC assertion — not the raw secret.
2. **Never weaken the path validator.** `bridge-server/path-validator.js` is the primary defense against path traversal. Do not simplify the ancestor-`realpathSync` + `path.relative()` tail algorithm. Any change must preserve the symlink-safe guarantee.
3. **Never skip `crypto.timingSafeEqual` for HMAC comparison.** Naive string comparison leaks secret length via timing side-channel.
4. **Never use `pm2 reload` for the bridge.** Use `pm2 restart`. `pm2 reload` creates a rolling-restart window where two generations of the bridge share the same `controllerEpoch`, which breaks the security model.
5. **Never commit `ecosystem.config.local.js`.** It is gitignored and contains secrets.
6. **Never add `shell: true` to `child_process.spawn` calls.** The bridge spawns `claude` without a shell by design.

### Protocol versioning

The message envelope `{ v: 1, type, seq, payload }` is versioned. Protocol v1 is frozen after Phase 2. Do not rename, remove, or change the type of any field in an existing message type without bumping `v`. Add new optional fields freely; changing semantics of existing fields requires a version bump.

All bridge→client and client→bridge message schemas live in **both** `bridge-server/index.js` (validation comments) and `ui/lib/protocol.ts` (Zod schemas). Keep them in sync.

### Session state machine

The session FSM in `session.js` has exactly four states: `IDLE`, `RUNNING`, `AWAITING_APPROVAL`, `DISCONNECTED`. The transitions are documented in `docs/PLAN.md` and `docs/IMPLEMENTATION_LOG.md`. Do not add states without updating both documents and the `state_sync` message payload.

Key invariant: **never auto-deny a pending approval on WebSocket drop.** `preDisconnectState` is saved; the TTL-expiry timer fires DENY only if `state === "DISCONNECTED"` when it fires.

### `controllerEpoch`

`controllerEpoch` is a SessionManager-scoped UUID, not per-session. It is minted once per authentication handshake. All mutating C→S messages must carry the current epoch; a mismatch results in `ws.close(4403)`. Do not store the epoch per-session.

## Code style

### Bridge server (`bridge-server/`)

- **Plain CommonJS** — no TypeScript, no transpilation, no build step. The bridge runs directly with `node`.
- JSDoc annotations are welcome; TypeScript-specific syntax (type casts, generics in `.js` files) is not valid and will cause `node --test` to fail.
- Use `const` / `let`, never `var`.
- Single-file modules — avoid splitting into many small files. The bridge has no dependency injection framework; inter-module coupling uses Node's module cache (singleton pattern).
- Error handling: use `if (!x) throw new Error('...')` for invariant violations. Don't swallow errors silently.
- Logging: use `bridge-server/logger.js`. Never `console.log` in bridge-server code; use the logger's structured API.

### Next.js UI (`ui/`)

- TypeScript. All component props must have explicit types.
- Zod schemas in `ui/lib/protocol.ts` are the single source of truth for all message types. Add new types there first, then use the inferred TypeScript types in components.
- `useWebSocket.ts` is a single hook — do not split it into sub-hooks without good reason. The connection lifecycle has subtle ordering dependencies.
- `export const dynamic = "force-dynamic"` must remain on `/api/token/route.ts`. Removing it would allow Next.js to statically cache the response, causing assertion replay.
- No `window` access in SSR paths. Guard with `typeof window !== "undefined"` or use `useEffect`.

## Development workflow

### Running locally (no Tailscale required)

```bash
# Start stub + UI
./bin/dev

# Or manually:
node bridge-server/stub.js &   # canned sequence stub, no auth
cd ui && npm run dev            # Next.js dev server at localhost:3000
```

The stub plays back a fixed canned sequence including all warning subtypes and an approval flow. Use it for UI development.

### Running tests

```bash
# All bridge unit tests (Node's built-in test runner)
cd bridge-server && node --test test/**/*.test.js

# Single suite
node --test test/auth.test.js

# Tailnet integration tests (requires Tailscale + real device)
./bin/test-tailnet
./bin/test-tailnet --skip-manual   # automated checks only
```

Tests use Node's built-in `node:test` and `node:assert`. No Jest, no Mocha, no additional test runner packages.

### Installing dependencies

```bash
cd bridge-server && npm install   # installs better-sqlite3, re2 (native addons)
cd ui && npm install && npm run build
```

After installing `bridge-server` deps, verify native addons:

```bash
node -e "require('re2'); console.log('re2 OK')"
node -e "require('better-sqlite3')(':memory:'); console.log('sqlite OK')"
```

If `re2` fails to build, the fallback (50ms timeout on native RegExp) is automatic — safe to continue, but note it in your work.

### Starting production processes

```bash
pm2 start ecosystem.config.js    # start all three processes
pm2 status                        # verify remotedev-bridge, remotedev-ui, remotedev-tailscale-serve
pm2 logs remotedev-bridge         # bridge logs
./bin/reload                      # restart with updated env (e.g. after token rotation)
```

## Dependency constraints

| Package | Why it exists | Notes |
|---|---|---|
| `better-sqlite3` | SQLite persistence (WAL mode, synchronous API) | Native addon — requires C++ toolchain |
| `re2` | ReDoS-safe regex for path validation | Native addon — fallback is automatic |
| `ws` | WebSocket server | Do not replace with a higher-level abstraction |
| `zod` (UI only) | Runtime schema validation + TypeScript inference | Single source of truth for protocol types |

Do not add new runtime dependencies to `bridge-server/` without strong justification. The bridge has no framework; keep it that way. The UI can use npm packages normally within reason.

Node.js version: **22 LTS** (pinned in `.nvmrc`). `better-sqlite3` does not have prebuilt binaries for Node 25+.

## Working with the session FSM

When making changes to `session.js`, keep these in mind:

- **`destroyed` guard:** Every async callback in `session.js` starts with `if (this.destroyed) return`. Add this guard to any new async operation.
- **`endSession(reason)` is the single teardown path.** It is idempotent (sets `this.destroyed = true` on first call). Never tear down a session through any other code path.
- **Backpressure:** `ws.close(1011)` evicts the connection but leaves `state = RUNNING`. The client reconnects via `resume_session`. Do not change this — closing the session on backpressure would lose in-flight `claude` output.
- **`caffeinate -i`** is spawned per session and `SIGKILL`'d in `endSession()`. If you refactor the child process setup, make sure `caffeinate` is still killed.

## What to document

When making a non-trivial change:

1. **`docs/IMPLEMENTATION_LOG.md`** — Add a section describing what changed, why, and any design decisions or bugs found. Use the existing entries as a template.
2. **`docs/TODO.md`** — Update the status checklist (✅/🔄/⬜/❌) to reflect completed or newly discovered work.
3. If you change a protocol message type, update **both** `bridge-server/index.js` and `ui/lib/protocol.ts`.
4. **Document version and changelog** — When editing any document (e.g. under `docs/`, or any `.md` file): add a version line at the top (e.g. `Version: 1`) if absent, or increment the existing version; and append a changelog entry at the end of the document with the new version number, date, and a short description of the change.

## Common pitfalls

- **`Map.prototype.values()` returns a `MapIterator`, not an array.** Spread it before calling `.filter()`/`.map()`: `[...map.values()].filter(...)`. This bit `activePids()` in `session-manager.js` during initial implementation.
- **TypeScript syntax in `.js` files.** `path-validator.test.js` had `null as unknown as string` (TS cast syntax) — not valid in CommonJS. Use plain JS equivalents.
- **`ecosystem.config.local.js` is a JS file, not a shell script.** Do not try to `source` it; `require()` it or let pm2 load it via `merge_defaults`.
- **`pm2 reload` vs `pm2 restart`.** Always `pm2 restart` for the bridge. See security rules above.
- **Next.js static export.** Do not add `output: "export"` to `next.config.js`. The `/api/token` route requires server-side execution.
- **`force-dynamic` on token route.** If you add caching or ISR to `/api/token/route.ts`, assertions will be replayed — a security bug.

## Phase gate (don't skip)

Before deploying to a real device, the following must pass:

1. `cd bridge-server && node --test test/**/*.test.js` — all 42 tests green
2. `cd ui && npm run build` — no TypeScript errors, no build failures
3. `./bin/test-tailnet --skip-manual` — automated tailnet checks pass
4. Phase 1a smoke gate on real iPhone (see `docs/TODO.md`)
